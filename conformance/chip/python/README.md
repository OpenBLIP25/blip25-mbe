# DVSI USB-3000 / AMBE-3000R driver

Python driver for the DVSI USB-3000 board (with AMBE-3000R chip) used as
a reference for blip25-mbe synthesis debugging.

## Hardware setup

The chip lives on the home Proxmox host (`pve`, 192.168.1.6) at
`/dev/ttyUSB0`. It's an AMBE-3000R (Rate Converter variant) connected
via FTDI FT231X USB-to-UART bridge (USB 0403:6015) at 460,800 baud.

Despite the "RC" name, the chip supports P25 full-rate IMBE
(7200 bps with FEC) — this is the variant that ships in the USB-3000.

## Usage

Copy to the host with the chip:

```bash
scp dvsi_driver.py root@pve:/usr/local/bin/dvsi
```

Then on the chip host:

```bash
# Verify chip responds
python3 dvsi_driver.py version
# → "0AMBE3000R V120.E100.XXXX.C106.G514.R009.B0010411.C0020208"

# Decode a P25 full-rate FEC bitstream to PCM
python3 dvsi_driver.py decode clean.bit clean_chip.pcm

# Compare chip decode against a reference PCM
python3 dvsi_driver.py compare clean.bit clean_ref.pcm --frames 60
```

## Why the chip's output may differ from DVSI test vectors

The DVSI test vectors (e.g. `tv-rc/p25/clean.pcm`) were generated by
the DVSI tool with specific encoder settings (noise suppression,
echo cancellation, gain) and chip state. Decoding `clean.bit` on a
fresh chip produces a similar but not bit-exact result because the
chip's predictor state, noise generator, and OLA buffer evolve from
cold start in a way that depends on the exact frame sequence.

For synthesis debugging we use the chip's **live** output as the
reference — feed the chip an arbitrary bitstream and treat its PCM
output as ground truth, regardless of whether it matches any prior
test vector.

## Critical safety: never send PKT_RESET

Sending the `PKT_RESET` field (`0x33`) hangs the AMBE chip until
physical USB unplug/replug. The Proxmox host's xhci_hcd root hub
does NOT support per-port power switching, so software cannot
recover. Use `PKT_INIT` (`0x0B`) for soft initialization instead.

This driver intentionally does not expose PKT_RESET.

## Recovery if the chip becomes unresponsive

If the FTDI device disappears (e.g., after `echo 1 > .../remove`):

```bash
# On pve, cycle the entire usb3 hub root
echo 0 > /sys/bus/usb/devices/usb3/authorized
sleep 2
echo 1 > /sys/bus/usb/devices/usb3/authorized
```

This recovers the FTDI side. If the AMBE chip is hung (e.g., after
PKT_RESET), the only fix is physical unplug + replug.

## Protocol notes

See module docstring in `dvsi_driver.py` for the full packet format
documentation. The key references are:

- AMBE-3000R Vocoder Chip User's Manual (v1.4, March 2013) §6 — packet
  protocol, all field IDs and formats
- USB-3000 User's Manual (v2.6) §4.5 + Tables 8-9 — P25 RATEP config
- DVSI cmpp25.txt — verified rate config words for P25 modes
